# 智能影视信息管理系统 PRD

## 1. 项目概述

### 1.1 项目名称

智能影视信息管理系统

### 1.2 项目背景

随着互联网影视资源数量的不断增长，用户在查找、收藏、评价和管理影视作品时面临信息分散、筛选效率低、个性化推荐不足等问题。传统影视信息管理系统通常只提供简单的影片增删改查功能，缺少用户行为分析、个性化推荐、数据统计和自动化信息导入能力。

本系统拟设计并实现一个基于 Web 的智能影视信息管理系统，面向普通用户和后台管理员，提供影视信息管理、用户收藏、评分评论、观看记录、待看片单、影视推荐、数据统计等功能。系统通过接入 TMDB 影视数据接口，实现影视基础信息的自动导入，减少人工录入成本；通过协同过滤算法，根据用户评分、收藏、浏览和观看行为生成个性化影视推荐，提高系统的智能化程度。

### 1.3 项目目标

本系统的主要目标如下：

1. 实现影视作品、演员、导演、分类、标签等基础信息的管理。
2. 实现用户注册、登录、个人资料维护、收藏、评分、评论、观看记录和待看片单功能。
3. 实现管理员后台管理功能，包括影片管理、用户管理、评论管理、公告管理、数据统计等。
4. 接入 TMDB API，支持根据影片名称或 TMDB ID 自动导入影视基础数据。
5. 基于用户行为数据实现协同过滤推荐，为用户提供个性化影视推荐。
6. 使用 ECharts 实现影视类型分布、评分分布、用户增长、收藏排行等数据可视化。
7. 支持可选 Redis 缓存热门影视、排行榜、首页推荐等高频访问数据，在未启用 Redis 时系统仍可正常运行。
8. 形成一个功能完整、结构清晰、可演示、可扩展的课程设计项目。

## 2. 项目定位

### 2.1 系统类型

本系统属于综合型 Web 信息管理系统，兼具以下特点：

* 影视信息管理系统
* 用户行为记录系统
* 个性化推荐系统
* 后台数据统计系统
* 第三方 API 数据导入系统

### 2.2 目标用户

系统主要面向两类用户：

#### 普通用户

普通用户可以浏览影视作品、搜索影片、查看详情、收藏影片、评分评论、记录已观看影片、维护待看片单，并接收系统推荐的影视作品。

#### 管理员

管理员可以登录后台，对影视数据、用户数据、评论数据、公告数据和推荐数据进行管理，并查看系统统计信息。

### 2.3 项目特色

本系统区别于普通影视 CRUD 管理系统的特点包括：

1. 支持 TMDB API 自动导入影视信息。
2. 支持用户收藏、评分、评论、浏览记录和观看记录。
3. 支持基于协同过滤算法的个性化推荐。
4. 支持后台数据统计和可视化图表展示。
5. 支持可选 Redis 缓存热门数据，在启用时提高访问效率。
6. 支持影视作品与演员、导演、标签之间的多对多关系设计，数据库结构更完整。

## 5. 功能模块设计

## 5.1 用户认证模块

### 5.1.1 用户注册

#### 功能描述

用户可以通过用户名、密码、昵称、邮箱完成注册。

#### 输入字段

| 字段              | 说明   |
| --------------- | ---- |
| username        | 用户名  |
| password        | 密码   |
| confirmPassword | 确认密码 |
| nickname        | 昵称   |
| email           | 邮箱   |

#### 业务规则

1. 用户名不能为空。
2. 密码不能为空，长度不少于 6 位。
3. 两次输入密码必须一致。
4. 用户名不能重复。
5. 邮箱格式必须正确。
6. 注册成功后默认角色为普通用户。

#### 输出结果

注册成功后返回提示信息，用户可跳转到登录页面。

---

### 5.1.2 用户登录

#### 功能描述

用户输入用户名和密码进行登录，登录成功后系统返回 JWT Token。

#### 输入字段

| 字段       | 说明  |
| -------- | --- |
| username | 用户名 |
| password | 密码  |

#### 业务规则

1. 用户名和密码不能为空。
2. 密码错误时提示登录失败。
3. 用户状态为禁用时禁止登录。
4. 登录成功后记录登录日志。
5. 根据用户角色跳转到普通用户页面或后台管理页面。

#### 输出结果

返回用户基本信息、角色信息和 Token。

---

### 5.1.3 修改密码

#### 功能描述

登录用户可以修改自己的密码。

#### 输入字段

| 字段              | 说明    |
| --------------- | ----- |
| oldPassword     | 原密码   |
| newPassword     | 新密码   |
| confirmPassword | 确认新密码 |

#### 业务规则

1. 原密码必须正确。
2. 新密码长度不少于 6 位。
3. 新密码和确认密码必须一致。
4. 修改成功后需要重新登录。

## 5.4 TMDB 数据导入模块

### 5.4.1 影片搜索导入

#### 功能描述

管理员可以在后台输入影片名称，通过 TMDB API 搜索影片信息，并选择目标影片导入系统数据库。

#### 操作流程

1. 管理员进入“影片导入”页面。
2. 输入影片名称。
3. 后端调用 TMDB API 搜索。
4. 前端展示搜索结果。
5. 管理员选择需要导入的影片。
6. 系统获取影片详情、类型、演员、导演、海报等信息。
7. 系统保存到本地 MySQL 数据库。

#### 导入字段

| 字段               | 说明         |
| ---------------- | ---------- |
| tmdbId           | TMDB 影片 ID |
| title            | 中文标题或本地化标题 |
| originalTitle    | 原标题        |
| overview         | 简介         |
| posterPath       | 海报路径       |
| backdropPath     | 背景图路径      |
| releaseDate      | 上映日期       |
| runtime          | 时长         |
| voteAverage      | TMDB 平均评分  |
| voteCount        | TMDB 评分人数  |
| originalLanguage | 原始语言       |
| genres           | 类型         |
| actors           | 演员         |
| directors        | 导演         |

#### 业务规则

1. 同一个 TMDB ID 不能重复导入。
2. 导入时优先保存影片基础信息。
3. 演员、导演不存在时自动新增。
4. 类型不存在时自动新增。
5. 影片与演员、导演、类型建立关联关系。
6. 如果 TMDB API 请求失败，应给出错误提示。
7. 导入成功后，影片进入本地影视库，后续查询走本地数据库。

## 5.8 个性化推荐模块

### 5.8.1 推荐目标

根据用户历史行为，为用户推荐可能感兴趣的影视作品。

### 5.8.2 数据来源

推荐算法使用以下用户行为数据：

| 行为    | 权重建议 |
| ----- | ---- |
| 用户评分  | 最高   |
| 用户收藏  | 较高   |
| 标记已观看 | 中等   |
| 加入待看  | 中等   |
| 浏览详情  | 较低   |

### 5.8.3 推荐方式

系统可采用基于用户的协同过滤算法。

#### 基本思想

如果用户 A 和用户 B 对多部影视作品的评分、收藏或观看行为相似，则认为用户 A 和用户 B 兴趣相近。系统可以将用户 B 喜欢而用户 A 尚未观看或收藏的影视作品推荐给用户 A。

### 5.8.4 算法流程

1. 收集用户行为数据。
2. 构建用户-影片评分矩阵。
3. 计算用户之间的相似度。
4. 找出与当前用户最相似的若干用户。
5. 获取相似用户喜欢但当前用户未接触过的影片。
6. 计算推荐分数。
7. 按推荐分数排序。
8. 生成推荐列表。
9. 将推荐结果保存到数据库。
10. 前端展示推荐结果。

### 5.8.5 相似度计算

可以使用余弦相似度。

公式说明：

设用户 A 和用户 B 的评分向量分别为 A 和 B，则相似度为：

sim(A, B) = A · B / (|A| × |B|)

相似度越接近 1，表示两个用户兴趣越相似。

### 5.8.6 推荐结果展示

用户登录后，可以在以下位置看到推荐结果：

1. 首页个性化推荐区域
2. 个人中心推荐页面
3. 影片详情页相似推荐区域

### 5.8.7 冷启动处理

当新用户行为数据不足时，系统采用以下推荐策略：

1. 推荐热门影片。
2. 推荐高评分影片。
3. 根据用户选择的兴趣标签推荐。
4. 根据最新上映影片推荐。

## 9. Redis 设计

### 9.1 Redis 使用场景

Redis 为可选缓存组件；启用时用于缓存高频访问数据，提高系统响应速度，未启用时系统回退到 MySQL 聚合与查询。

### 9.2 缓存内容

| Key                  | 内容     | 过期时间  |
| -------------------- | ------ | ----- |
| movie:hot            | 热门影视列表 | 30 分钟 |
| movie:top-rated      | 高分影视列表 | 30 分钟 |
| movie:latest         | 最新影视列表 | 30 分钟 |
| movie:detail:{id}    | 影视详情   | 10 分钟 |
| recommend:user:{id}  | 用户推荐列表 | 30 分钟 |
| statistics:dashboard | 后台首页统计 | 10 分钟 |

### 9.3 缓存更新策略

1. 启用 Redis 时查询优先读 Redis。
2. Redis 未命中或未启用时查询 MySQL。
3. 启用 Redis 时，查询 MySQL 后写入 Redis。
4. 影片、评分、收藏等数据更新后删除相关缓存。
5. 下一次查询时重新生成缓存。

## 11. 非功能需求

### 11.1 性能需求

1. 首页加载时间应控制在 3 秒以内。
2. 普通分页查询响应时间应控制在 1 秒以内。
3. 热门影视、排行榜等高频数据在启用 Redis 时使用缓存。
4. 后台统计数据可采用 Redis 缓存；若未启用 Redis，则直接执行聚合查询。

### 11.2 安全需求

1. 用户密码必须加密存储。
2. 登录状态使用 JWT 管理。
3. 后台接口必须进行管理员权限校验。
4. 用户只能修改自己的个人信息。
5. 普通用户不能删除他人评论。
6. 对用户输入内容进行基本校验，防止恶意输入。
7. 删除重要数据时采用逻辑删除。

### 11.3 易用性需求

1. 前台页面风格简洁清晰。
2. 后台使用 Element Plus 表格、表单、弹窗等组件，提高操作效率。
3. 列表页面支持搜索、筛选、分页。
4. 操作成功或失败时提供明确提示。
5. 重要操作如删除需要二次确认。

### 11.4 可维护性需求

1. 前后端分离开发。
2. 后端按照 Controller、Service、Mapper、Entity 分层。
3. 前端按照页面、组件、接口请求、状态管理分层。
4. 数据库表命名规范。
5. 接口返回格式统一。
6. 关键业务逻辑添加注释。

## 16. 项目风险与解决方案

### 16.1 TMDB API 请求失败

风险：网络不稳定或 API 请求失败可能影响导入功能。

解决方案：

1. 导入后的数据保存到本地 MySQL。
2. 系统主要功能依赖本地数据库，不依赖实时 API。
3. 准备部分本地测试数据，保证答辩时可正常演示。

### 16.2 推荐算法数据不足

风险：用户行为数据较少时，协同过滤推荐效果不明显。

解决方案：

1. 提前准备测试用户和评分数据。
2. 新用户默认推荐热门影片。
3. 行为不足时结合高分榜和热门榜推荐。

### 16.3 功能过多导致开发时间不足

风险：模块过多可能导致开发进度紧张。

解决方案：

1. 优先完成核心功能。
2. 将功能分为必须完成和扩展完成。
3. 推荐算法、Redis 缓存、数据统计作为重点亮点优先实现。

### 16.4 前后端联调复杂

风险：接口数量较多，联调容易出现问题。

解决方案：

1. 使用统一返回格式。
2. 使用接口文档工具。
3. 前端 Axios 统一封装请求。
4. 后端统一异常处理。

## 18. 项目总结

智能影视信息管理系统以影视作品信息管理为核心，结合用户行为记录、个性化推荐、第三方数据导入和后台数据统计，实现了一个较完整的 Web 信息管理系统。系统采用 Vue 3、Element Plus、CSS 变量与普通 CSS 构建前端页面，采用 Spring Boot、MyBatis-Plus、MySQL、Redis 构建后端服务，整体技术路线成熟，适合课程设计实现。

系统不仅包含传统管理系统所需的增删改查、分页查询、权限控制和数据统计功能，还加入了 TMDB API 自动导入和协同过滤推荐算法，使系统具备一定智能化特征，能够体现“智能信息系统综合课程设计”的课程要求。
